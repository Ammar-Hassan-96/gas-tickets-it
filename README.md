# GAS Internal Tickets — German Auto Service
## نظام التيكتات الداخلي بين الإدارات · v2 (نموذج هرمي)

نظام كامل للطلبات الداخلية بين إدارات الشركة، **بعزل كامل بين الإدارات** — كل إدارة تشوف شغلها فقط.

---

## 🎯 اللي اتعمل في هذه النسخة (v2)

بالإضافة لكل ما جاء في v1 (القوائم المرتبطة، المرفقات، routing الإشعارات)، **v2 تضيف طبقة حماية هرمية**:

- 👑 **4 أدوار بطبقات واضحة**: super_admin → manager → supervisor → employee
- 🔒 **عزل كامل بين الإدارات**: كل إدارة تشوف طلباتها فقط
- 🎛️ **صفحة "الأدوار"**: لوحة مركزية للـ super_admin لتعيين دور وإدارة كل موظف دفعة واحدة
- 👁️ **شفافية ذكية**: مقدم الطلب يشوف كل تفاصيل طلبه حتى لو من إدارة تانية
- 🛡️ **فلترة على مستوى الـ UI**: `visibleTickets()` و `visibleUsers()` helpers مركزية

## 🏗️ نموذج الأدوار والصلاحيات

```
👑 Super Admin (ammar.admin)
   └── يشوف كل حاجة · يدير النظام · يعين الأدوار · يدير الإدارات

🏢 Manager (مدير إدارة — واحدة فقط)
   ├── يشوف طلبات إدارته + طلباته الشخصية
   ├── يدير موظفي إدارته (إضافة/تعديل/تعيين كلمة مرور)
   └── يدير الأرشيف لإدارته

⭐ Supervisor (مشرف إدارة)
   ├── يشوف طلبات إدارته + طلباته الشخصية
   ├── يعين الطلبات لموظفي إدارته
   └── يقدر يحلها بنفسه

👤 Employee (موظف)
   ├── يشوف طلباته الشخصية (في أي إدارة قدمها)
   ├── يشوف الطلبات المعينة ليه شخصياً
   └── يشوف الطلبات المفتوحة (open) في إدارته — عشان يستلمها
```

### قواعد الرؤية

| المستخدم | يشوف |
|---|---|
| **Super Admin** | كل الطلبات · كل المستخدمين · كل الإدارات |
| **Manager** (إدارة X) | كل طلبات X الموجهة له + طلباته الشخصية + موظفي X |
| **Supervisor** (إدارة X) | كل طلبات X الموجهة له + طلباته الشخصية |
| **Employee** (إدارة X) | طلباته الشخصية + الطلبات المعينة عليه + طلبات X المفتوحة (open) |

### قواعد التعيين والفعل

| من يقدر يعين؟ | من يقدر يفعل/يرد؟ | من يقدر يحذف/يؤرشف؟ |
|---|---|---|
| أي موظف في نفس إدارة الطلب | المعين عليه + قيادة الإدارة | مدير الإدارة + super_admin |

### شفافية مقدم الطلب
بغض النظر عن إدارته، مقدم الطلب **دايماً** يشوف:
- تفاصيل طلبه كاملة
- اسم اللي متعين عليه
- كل التعليقات والردود
- حالة الطلب الحالية

## 🚀 خطوات التفعيل (3 خطوات)

### 1️⃣ تشغيل الـ Migrations في Supabase

افتح **Supabase → SQL Editor → New query** وشغّل الملفين بالترتيب:

**أولاً:** محتوى `supabase-migration-internal.sql`
**ثانياً:** محتوى `supabase-migration-roles-v2.sql`

الـ migrations كلها آمنة — بتضيف بس ومش بتحذف أي بيانات.

### 2️⃣ رفع على GitHub → Netlify
زي العادة — push وخلاص.

### 3️⃣ تحديد الأدوار والإدارات (من التطبيق)

بعد أول login كـ `ammar.admin`:

1. هتلاقي في الـ navigation **"🔑 الأدوار"** — اضغط عليها
2. هتشوف:
   - **جدول نظرة عامة** لكل إدارة (عدد المديرين والمشرفين والموظفين)
   - **جدول المستخدمين** — كل واحد قدامه dropdown للدور و dropdown للإدارة
3. حدّد لكل موظف الدور المناسب والإدارة
4. اضغط **"💾 حفظ كل التغييرات"** في آخر الصفحة

✅ خلاص — النظام جاهز.

## 🎭 سيناريوهات عملية

### مثال 1: موظف الحسابات يطلب فاتورة
- **أحمد** (employee في الخدمة) يفتح طلب → إدارة الحسابات → نوع: "طلب فاتورة"
- إشعار يروح لـ **مديرين ومشرفين إدارة الحسابات** فقط
- **سارة** (employee في الحسابات) لو فتحت "طلبات إدارتها" هتلاقي الطلب (لأنه open)، تقدر تستلمه
- بمجرد ما سارة استلمته، **موظفي الحسابات التانيين** مش هيشوفوه (لأنه بقى in_progress)
- بس **المدير والمشرف** يفضلوا شايفينه
- **أحمد** (مقدم الطلب) دايماً يشوف حالته والتعليقات حتى لو من إدارة تانية

### مثال 2: موظف المبيعات يطلب IT support
- طلب IT يروح لإدارة IT بس
- موظفي المبيعات (حتى مدير المبيعات) مش يشوفوا الطلب
- أحمد نفسه يشوف تفاصيل طلبه و اللي متعين عليه

### مثال 3: مدير الحسابات يبص على فريقه
- يفتح **"التقارير"** → يشوف أداء موظفي الحسابات فقط
- يفتح **"المستخدمون"** → يشوف موظفي الحسابات فقط
- **مش** هيشوف أي بيانات عن الإدارات التانية

## 📝 ملاحظات تقنية

### طبقة Perm المركزية
كل قواعد الصلاحيات مجمعة في object واحد اسمه `Perm` في بداية `app.js`. أي تعديل مستقبلي على الصلاحيات بيتم من هنا فقط.

### التوافق مع البيانات القديمة
- الدور القديم `admin` يُعامل كـ `supervisor` (legacy)
- التيكتات القديمة بدون `target_department` تظهر للـ admins/managers القدامى + super_admin
- البيانات كلها محفوظة وآمنة

### الأمان
- **فلترة على مستوى UI**: الموظف مش يشوف بيانات مش صلاحياته
- **حماية backend**: العمليات الحساسة (حذف/إعادة تعيين كلمة سر) محمية في `auth.mjs`
- **حساب `ammar.admin` محمي**: لا يمكن حذفه أو تعديل دوره من الواجهة

### ملاحظة مهمة على `admin` القديم
إذا عندك مستخدمين بدور `admin` من النسخة القديمة، هيفضلوا شغالين عادي بصلاحيات supervisor. لكن الأحسن تحوّلهم لـ `supervisor` أو `manager` من صفحة "الأدوار" عشان الوضوح.

## 🛠️ استكشاف الأخطاء

**صفحة "الأدوار" مش ظاهرة؟**
تأكد إنك داخل بحساب `ammar.admin`، وإن الـ migration v2 شغّل:
```sql
SELECT role, count(*) FROM users GROUP BY role;
-- لازم يكون في super_admin واحد على الأقل
```

**موظف مش شايف طلب إدارته المفتوح؟**
تأكد إن `department` بتاعه مطابق تماماً لاسم الإدارة المستهدفة (case-sensitive):
```sql
SELECT id, name, department, role FROM users WHERE is_active = true;
```

**الإشعارات مش بتوصل؟**
الإشعارات الجديدة بتروح للـ `manager` و `supervisor` في الإدارة المستهدفة فقط (مش الموظفين العاديين). لو الإدارة مفيهاش قيادة، الإشعار بيروح للـ super_admins.
