## 🏗️ نموذج الأدوار

```
👑 Super Admin (ammar.admin)
   └── كل الصلاحيات · يرى كل شيء · يعين الأدوار

🏢 Manager (مدير إدارة — واحدة)
   ├── يرى طلبات إدارته + طلباته الشخصية
   ├── يعين · يحدث · يحذف طلبات إدارته
   ├── يدير موظفي إدارته (تعديل · حذف · إعادة تعيين كلمة سر)
   └── يرى تقارير فريقه

⭐ Supervisor (مشرف إدارة)
   ├── يرى طلبات إدارته + طلباته الشخصية
   ├── يعين · يحدث طلبات إدارته
   └── ما يقدرش يحذف تيكتات أو موظفين

👤 Employee (موظف)
   ├── يرى طلباته الشخصية (في أي إدارة)
   ├── يرى الطلبات المعينة عليه
   └── يرى طلبات إدارته المفتوحة (open) عشان يستلمها
```

## 🎭 سيناريوهات عملية

### مدير الحسابات يحذف موظف من فريقه
- يروح "المستخدمون" → هيشوف موظفي الحسابات بس
- يضغط "حذف" على الموظف → الـ server يتحقق:
  - هو manager؟ ✅
  - الموظف في نفس إدارته؟ ✅
  - الموظف مش manager أو super_admin؟ ✅
- ← تم الحذف

### موظف IT يستلم طلب صيانة جهاز
- يفتح "الرئيسية" → يشوف طلبات IT المفتوحة
- يضغط على طلب → صفحة التفاصيل تفتح
- "تعيين لي" → التيكت بقى عليه
- دلوقتي موظفي IT التانيين **ما هيشوفوش التيكت** (صار in_progress)
- بس المدير والمشرف يفضلوا شايفينه

### مقدم طلب من المبيعات يتابع طلبه للـ IT
- يفتح "طلباتي" → هيشوف طلبه
- يضغط عليه → **يشوف كل شيء**:
  - اسم المعين عليه
  - كل التعليقات
  - الحالة الحالية
  - SLA
- ممكن يرد بتعليق → المعين عليه بيتبلّغ

## 🛠️ استكشاف الأخطاء

### "ليس لديك صلاحية لعرض هذا الطلب"
- تأكد إن الـ deploy الأخير نزل ✅ (اتصلحت في v2.2)
- مسح cache المتصفح (Ctrl+Shift+R)

### "كل الطلبات" فاضي للمدير/المشرف
- 99% المشكلة إن `department` عنده مش مطابق لـ `target_department` على التيكتات
- شغّل `diagnostic.sql` في Supabase
- أو شغّل `supabase-migration-v3-data-fixes.sql` — هيصلح المسافات الزائدة تلقائياً
- ادخل "🔑 الأدوار" وتأكد إن كل موظف متعين له إدارة صح

### مدير الإدارة مش شايف زرار "حذف"
- بعد v2.2 المفروض يظهر
- تأكد إن الموظف في نفس إدارته (DB comparison)
- الموظف لازم يكون `employee` أو `supervisor` (المدير ما يقدرش يحذف مدير تاني)

### الإشعارات مش بتوصل
- الإشعارات بتروح لمديرين/مشرفين الإدارة المستهدفة بس
- لو الإدارة مفيهاش قيادة، بتروح لـ super_admin
- تأكد إن الـ `department` في users مطابق للـ `target_department` بالحروف الكبيرة/الصغيرة

## 📝 ملاحظات تقنية

### طبقة Perm المركزية
كل قواعد الصلاحيات في object اسمه `Perm` في بداية `app.js`. أي تعديل على الصلاحيات في المستقبل يحصل من هنا بس:
- `Perm.canSeeTicket(t)` — رؤية تيكت
- `Perm.canActOnTicket(t)` — تحديث/رد
- `Perm.canAssignTicket(t)` — تعيين
- `Perm.canDeleteTicket(t)` — حذف
- `Perm.canManageUser(u)` — تعديل مستخدم
- `Perm.canDeleteUser(u)` — حذف مستخدم
- `Perm.canResetUserPassword(u)` — إعادة تعيين كلمة مرور
- `Perm.canSeePage(page)` — رؤية صفحة

### الأمان الطبقي
- **Layer 1**: UI يخفي الأزرار حسب الصلاحية (`Perm.canXxx`)
- **Layer 2**: `visibleTickets()` و `visibleUsers()` يفلتروا البيانات قبل العرض
- **Layer 3**: `auth.mjs` يفحص الصلاحية مرة تانية server-side للعمليات الحساسة

حتى لو حد حاول يلعب في الـ UI من console، الـ server هيرفض العملية.

### الحساب المحمي
`ammar.admin` محمي من التعديل والحذف من أي حد غيره.
